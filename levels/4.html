Level 4 of 7

UI snipping

Don't re-engineer the fireplace — copy it. Production-grade components from 21st.dev, Magic UI, or CodePen drop straight in.

The technique

Component libraries publish battle-tested interactive pieces. Copy the code (or even just screenshots of it) and ask Claude to integrate. Here: a celebratory balloons animation (balloons-js).

The two models made genuinely different engineering choices from the same ask — Fable inlined the library for a self-contained file, Opus imported it from esm.sh at runtime. Same feature, different dependency philosophy.

Note: the source video does image/video at level 4 and UI snipping at level 5 — these runs did it in the reverse order.

The prompts that did it

Integrate the snipped component
Hey there. I'd like you to integrate those balloons feature at the bottom of the page. Make sure you keep the current index HTML file. Name it as yellow-iphone-web-level-4.html

Preceded by pasted screenshots (Fable) or the actual React/TSX source (Opus) of the balloons component. Opus translated React into vanilla JS on the fly.

Fix the trigger
The Bloom animation is supposed to be automatically shown up when hitting the end of the page, not only when I click on the button.

Both models rewired the trigger to an IntersectionObserver at page end. Fable went further and fixed an upstream memory leak — balloons-js never removes its wrapper elements, and Fable spotted the accumulating full-screen filters as a “dark smudge” in its own screenshots.

What changed vs level 3

  • A delightful, production-quality interaction appears with near-zero design risk.
  • Pasting a React component into a no-build HTML page just works — the model adapts the paradigm.
  • Model divergence shows up in dependency strategy: inline (Fable) vs CDN import (Opus).

What this level taught

  • The internet is a parts bin; snipping beats reinventing.
  • Models can find upstream library bugs during integration.
